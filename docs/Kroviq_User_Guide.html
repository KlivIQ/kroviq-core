Kroviq User Guide

Complete documentation for the Kroviq test automation platform. This guide covers installation, configuration, and daily usage of Kroviq Core and Kroviq Studio.

Installation

Purpose

Install Kroviq Core on your machine so you can build and run automated tests.

Prerequisites

Before you install Kroviq, make sure you have these tools on your computer:

ToolVersionWhy You Need It
Java JDK17 or higherKroviq is written in Java
Maven3.8 or higherBuilds and manages dependencies
IDEIntelliJ IDEA or EclipseWrite and edit code
GitAny recent versionClone the repository
Tip

Use IntelliJ IDEA Community Edition — it is free and works perfectly with Kroviq. Install the Lombok plugin from the IDE marketplace.

Setup Steps

When to Use

Follow these steps when you are setting up Kroviq for the first time on a new machine, or when you need to update to a newer version.

Step-by-Step Instructions

  1. Open a terminal (Command Prompt or PowerShell on Windows).
  2. Clone the Kroviq Core repository: 
    git clone https://github.com/your-org/Kroviq_Core.git
  3. Navigate into the project folder: 
    cd Kroviq_Core
  4. Build and install the framework: 
    mvn clean install
  5. Wait for the build to complete. You should see BUILD SUCCESS in the terminal.
  6. Open your test project and add the Kroviq dependency to your pom.xml: 
    <dependency>
    <groupId>com.kroviq</groupId>
    <artifactId>kroviq-core</artifactId>
    <version>2.0.0</version>
</dependency>
  7. Reload Maven in your IDE (right-click pom.xml → Maven → Reload Project).
Screenshot: Terminal showing BUILD SUCCESS after mvn clean install
The framework JAR is installed in your local Maven repository. Your test project can now use all Kroviq classes and utilities.
Tips

If you see SSL errors during Maven build, add these flags: -Dmaven.wagon.http.ssl.insecure=true -Dmaven.wagon.http.ssl.allowall=true

Note

Kroviq Core is a framework library — it does not contain test cases. You write your tests in a separate project (like Kroviq_TestSuiteTemplate) that depends on this framework.

Workspace

Purpose

Understand how the Kroviq workspace is organized so you know where to find and place files.

When to Use

Refer to this section when you are navigating the project for the first time, or when you need to find a specific file or folder.

Folder Structure

Kroviq_Framework/
├── Kroviq_Core/                   # Framework source code (do not modify)
├── Kroviq_TestSuiteTemplate/      # Your test project (write tests here)
│   ├── src/main/java/kroviq/
│   │   ├── constants/             # Element locators (XPaths, IDs)
│   │   ├── pages/                 # Page classes
│   │   └── helpers/               # Project-specific helpers
│   ├── src/test/java/
│   │   ├── runners/               # Test runner (entry point)
│   │   └── stepdefinitions/       # Cucumber step definitions
│   ├── src/test/resources/
│   │   └── features/              # Gherkin feature files
│   ├── TestDatastore/json/        # Test data files
│   ├── Reports/                   # Generated test reports
│   └── RunManager.json            # Execution configuration
└── kroviq-studio/                 # Kroviq Studio (optional UI platform)
Screenshot: IDE project explorer showing the workspace folder structure

Configuration

The main configuration file is RunManager.json in your test project root. It controls how tests are executed:

{
  "browser": "chrome",
  "environmentURL": "https://your-app.com",
  "executionMode": "LOCAL",
  "parallel": false,
  "threads": 1,
  "selectedSuite": "REGRESSION",
  "testDataSource": "json",
  "captureScreenshotsOnFailure": true
}
FieldWhat It DoesOptions
browserWhich browser to usechrome, firefox, edge
environmentURLBase URL of your applicationAny valid URL
executionModeWhere to run testsLOCAL, GRID, CLOUD
parallelRun tests at the same timetrue, false
threadsHow many tests run together1, 2, 3, etc.
selectedSuiteWhich test suite to runREGRESSION, SMOKE, etc.
testDataSourceWhere test data comes fromjson, excel
You understand the workspace layout and can locate any file you need. RunManager.json is configured for your environment.
Warning

Never modify files inside Kroviq_Core/ directly. It is a shared framework. All your test code goes in the test project (e.g., Kroviq_TestSuiteTemplate).

Projects

Purpose

A Project is the top-level container in Kroviq Studio. It represents one application or system that you are testing. All modules, pages, test cases, and execution profiles belong to a project.

When to Use

Create a project when you start testing a new application. Each application you test should have its own project.

Step-by-Step Instructions

Create a New Project

  1. Open Kroviq Studio in your browser (http://localhost:3000).
  2. Click Projects in the left sidebar.
  3. Click the + New Project button in the top-right corner.
  4. Fill in the form:
    • Name — A short name for your application (e.g., "WingIt Airlines")
    • Description — What this application does (optional)
    • Test Suite Path — Full path to your test project folder (e.g., D:/Kroviq_Framework/Kroviq_TestSuiteTemplate)
    • Framework Path — Path to Kroviq Core (e.g., D:/Kroviq_Framework/Kroviq_Core)
    • Framework Version — The version you are using (e.g., 2.0.0)
  5. Click Save.
Screenshot: New Project dialog with fields filled in
The project appears in the projects list. You can now create modules, pages, and test cases inside it.

Edit a Project

  1. Go to the Projects page.
  2. Click the three-dot menu (⋮) on the project card.
  3. Select Edit.
  4. Change the fields you need to update.
  5. Click Save.

Delete a Project

  1. Go to the Projects page.
  2. Click the three-dot menu (⋮) on the project card.
  3. Select Delete.
  4. Confirm the deletion in the dialog.
Warning

Deleting a project removes ALL its modules, pages, elements, test cases, and execution profiles. This action cannot be undone.

Tips

Use clear, short project names. Avoid special characters. The Test Suite Path must point to a valid folder that contains a pom.xml file.

Note

The Test Suite Path is required for execution. If you only want to design tests without running them, you can leave it empty initially and add it later.

Modules

Purpose

A Module represents a functional area of your application (e.g., Login, Flight Search, Booking, Payments). Modules help you organize your tests into logical groups.

When to Use

Create a module for each major feature or section of your application. Think of modules as chapters in a book — each one covers a different topic.

Step-by-Step Instructions

Create a New Module

  1. Select your project from the Projects page.
  2. Click Modules in the left sidebar.
  3. Click the + New Module button.
  4. Fill in the form:
    • Name — The module name (e.g., "Flight Search")
    • Module Tag — Auto-generated tag used for test execution (e.g., FLIGHT_SEARCH)
    • Description — What this module covers (optional)
  5. Click Save.
Screenshot: Modules page showing list of modules with tags
The module appears in the modules list. The module tag is automatically generated from the name (e.g., "FlightSearch" becomes "FLIGHT_SEARCH").

Understanding Module Tags

Module tags are used to:

  • Filter which tests to run (in execution profiles)
  • Organize feature files into folders
  • Tag test cases for suite selection

Tag generation rules:

Module NameGenerated Tag
LoginLOGIN
FlightSearchFLIGHT_SEARCH
DemoMUIDEMO_MUI
DemoPrimeNGDEMO_PRIME_NG

Edit a Module

  1. Go to the Modules page.
  2. Click the three-dot menu (⋮) on the module row.
  3. Select Edit.
  4. Update the fields.
  5. Click Save.

Delete a Module

  1. Click the three-dot menu (⋮) on the module row.
  2. Select Delete.
  3. Confirm the deletion.
Warning

Deleting a module removes all its pages, elements, and test cases. Make sure you have a backup if needed.

Tips

Keep module names short and descriptive. Use PascalCase (e.g., FlightSearch, not flight_search). One module per major feature is the recommended approach.

Note

Module tags are used in Cucumber tag expressions. When you run a suite, Kroviq filters tests by these tags. For example, running the "SMOKE" suite might include tests tagged with @SMOKE across multiple modules.

Pages

Purpose

A Page represents a screen or view in your application (e.g., Login Page, Search Results Page, Booking Confirmation Page). Pages group related elements together, following the Page Object Model pattern.

When to Use

Create a page for each distinct screen in your application. If a screen has many elements, you can split it into multiple pages (e.g., "Booking Form" and "Booking Summary" instead of one large "Booking" page).

Step-by-Step Instructions

Create a New Page

  1. Select your project and navigate to a module.
  2. Click Pages in the left sidebar.
  3. Click the + New Page button.
  4. Fill in the form:
    • Name — The page name (e.g., "LoginPage")
    • Group — Optional grouping for organization (e.g., "Authentication")
    • Description — What this page represents (optional)
  5. Click Save.
Screenshot: Pages list showing pages grouped by category with element counts
The page appears in the pages list. You can now add elements (locators) to this page in the Object Repository.

Organize Pages with Groups

Groups help you organize pages when a module has many screens:

GroupPages
AuthenticationLoginPage, RegisterPage, ForgotPasswordPage
SearchFlightSearchPage, SearchResultsPage
BookingPassengerDetailsPage, SeatSelectionPage, PaymentPage
Tips

Use PascalCase for page names and always end with "Page" (e.g., LoginPage, not login or Login_Page). This matches the Java class naming convention used in the generated Constants file.

Note

Pages are scoped to a module. You cannot share pages across modules. If two modules use the same screen, create the page in each module separately.

Object Repository

Purpose

The Object Repository stores all UI element locators (XPaths, CSS selectors, IDs) for your application. It is the single source of truth for how Kroviq finds elements on the screen.

When to Use

Use the Object Repository whenever you need to add, edit, or view element locators. Every element your tests interact with must be defined here first.

Step-by-Step Instructions

Add a New Element

  1. Navigate to Object Repository in the sidebar (under your active module).
  2. Select a page from the page tree on the left panel.
  3. Click the + Add Element button in the elements table.
  4. Fill in the form:
    • Constant Name — Java constant name (e.g., TXT_USERNAME)
    • Display Name — Human-readable name (e.g., "Username Input")
    • Locator Type — How to find the element (xpath, id, css, name)
    • Locator Value — The actual locator string (e.g., //input[@id='username'])
    • Action Type — What action to perform (CLICK, TYPE, SELECT, etc.)
  5. Click Save.
Screenshot: Object Repository split-pane view — page tree (left), elements table (center), Constants.java preview (right)
The element appears in the elements table. The Constants.java preview panel on the right updates to show the generated Java code.

Constants.java Preview

The right panel shows a live preview of the generated Java constants file:

public class LoginPageConstants {
    public static final String TXT_USERNAME = "//input[@id='username']";
    public static final String TXT_PASSWORD = "//input[@id='password']";
    public static final String BTN_LOGIN = "//button[@type='submit']";
    public static final String LBL_ERROR = "//div[@class='error-message']";
}

Edit an Element

  1. Click on the element row in the table.
  2. Update the fields in the edit form.
  3. Click Save.

Delete an Element

  1. Click the three-dot menu (⋮) on the element row.
  2. Select Delete.
  3. Confirm the deletion.
Tips

Prefer stable locators: use id, data-testid, or name attributes over complex XPaths. Shorter locators are faster and less likely to break when the UI changes.

Note

The Action Type determines how Kroviq interacts with the element. For example, a CLICK action will click the element, while a TYPE action will clear the field and type text into it. Choose the correct action type for each element.

Test Cases

Purpose

Test Cases define the steps your automated test will execute. Each test case is a sequence of actions (click, type, verify) that validates a specific behavior of your application.

When to Use

Create a test case for each scenario you want to automate. For example: "Successful Login", "Login with Invalid Password", "Search for One-Way Flight".

Step-by-Step Instructions

Create a New Test Case

  1. Navigate to Test Cases in the sidebar (under your active module).
  2. Click the + New Test Case button.
  3. Fill in the form:
    • Name — Descriptive name (e.g., "TC_LOGIN_001_ValidLogin")
    • Description — What this test validates (optional)
    • Tags — Cucumber tags for filtering (e.g., @SMOKE, @REGRESSION)
  4. Click Save.
Screenshot: Test Cases page with DesignerLayout — case list (left), step designer (center), test data panel (right)
The test case appears in the list. You can now add steps to it using the Step Designer.

Add Steps Using the Step Designer

  1. Select a test case from the list on the left.
  2. Click + Add Step in the Step Designer panel.
  3. Fill in the step dialog:
    • Keyword — Given, When, Then, And
    • Action — What to do (e.g., "click", "type", "verify text")
    • Object — Click the picker icon to open the Object Picker and select an element
    • Value — Data to use (e.g., text to type, or TD_Username for test data reference)
  4. Click Save Step.
  5. Repeat for each step in your test case.
Screenshot: Step Dialog showing keyword, action, object picker, and value fields

Using the Object Picker

  1. In the Step Dialog, click the picker icon next to the Object field.
  2. The Object Picker modal opens with a page tree on the left.
  3. Select a page from the tree.
  4. The right panel shows all elements on that page with a search bar.
  5. Click on the element you want to use.
  6. The modal closes and the element is linked to your step.
Screenshot: Object Picker modal — page tree (left) and element list with search (right)

Reorder Steps (Drag and Drop)

  1. Hover over a step row in the Step Designer.
  2. Click and hold the drag handle (≡) on the left side.
  3. Drag the step to its new position.
  4. Release to drop. The order is saved automatically.

Disable a Step

You can temporarily disable a step without deleting it:

  1. Click the three-dot menu (⋮) on the step row.
  2. Select Toggle Enabled.
  3. Disabled steps appear grayed out and are skipped during execution.
Tips

Use the TD_ prefix in the Value field to reference test data dynamically. For example, TD_Username will be replaced at runtime with the actual value from your test data file.

Note

Test case names should follow the convention: TC_MODULE_NNN_Description (e.g., TC_LOGIN_001_ValidLogin). This helps with organization and makes it easy to find tests in reports.

Test Data

Purpose

Test Data provides the values your tests use at runtime (usernames, passwords, search criteria, expected results). Separating data from test logic lets you run the same test with different inputs without changing code.

When to Use

Use test data whenever your test needs input values or expected results that might change between environments or test runs. Never hardcode values in your test steps.

Data Sources

Kroviq supports two data sources, controlled by the testDataSource field in RunManager.json:

SourceFormatBest For
JSONOne .json file per moduleMost projects (recommended)
ExcelOne .xlsx file per moduleTeams familiar with spreadsheets

Step-by-Step Instructions

Add Test Data in Studio

  1. Select a test case in the Test Cases page.
  2. Open the Test Data Panel on the right side.
  3. Click + Add Data.
  4. Enter a Key (e.g., "Username") and a Value (e.g., "john.doe@example.com").
  5. Click Save.
  6. In your test step, reference this data using TD_Username in the Value field.
Screenshot: Test Data Panel showing key-value pairs and JSON preview
The test data key-value pair is saved. At runtime, any step using TD_Username will automatically receive the value "john.doe@example.com".

JSON Test Data Format

JSON files are stored in TestDatastore/json/ with one file per module:

// TestDatastore/json/Login.json
{
  "TC_LOGIN_001_ValidLogin": {
    "Username": "john.doe@example.com",
    "Password": "SecurePass123",
    "ExpectedWelcome": "Welcome, John!"
  },
  "TC_LOGIN_002_InvalidPassword": {
    "Username": "john.doe@example.com",
    "Password": "WrongPassword",
    "ExpectedError": "Invalid credentials"
  }
}

How TD_ Resolution Works

When Kroviq encounters TD_FieldName in a step:

  1. It looks up the current test case name (e.g., "TC_LOGIN_001_ValidLogin")
  2. It finds the matching entry in the module's data file
  3. It replaces TD_FieldName with the actual value
Tips

Choose ONE data source per project and stick with it. If you need to migrate, use the built-in migration tools: ExcelToJsonMigrator or JsonToExcelMigrator.

Warning

Never store real passwords or sensitive data in test data files. Use placeholder values for local testing and environment variables for CI/CD pipelines.

Note

Test data is scoped to a module. Each module has its own data file. The TestDataManager automatically loads the correct file based on the current test's module tag.

Execution Profiles

Purpose

An Execution Profile defines HOW your tests run — which browser, which environment, how many threads, and which test suite. Profiles let you run the same tests in different configurations without editing files.

When to Use

Create execution profiles when you need to run tests in different environments (Dev, QA, Staging), different browsers (Chrome, Firefox, Edge), or with different parallelism settings.

Step-by-Step Instructions

Create a New Execution Profile

  1. Select your project.
  2. Click Execution Profiles in the sidebar.
  3. Click the + New Profile button.
  4. Fill in the form:
    • Name — Profile name (e.g., "Chrome QA Parallel")
    • Browser — chrome, firefox, or edge
    • Environment URL — Base URL of the target environment (e.g., https://qa.wingitairlines.com)
    • Execution Mode — LOCAL (your machine), GRID (Selenium Grid), or CLOUD (cloud provider)
    • Parallel — Enable or disable parallel execution
    • Threads — Number of parallel threads (1–10)
    • Selected Suite — Which test suite to run (e.g., REGRESSION, SMOKE)
    • Screenshots on Failure — Capture screenshots when a step fails
  5. Click Save.
Screenshot: Execution Profile form with all fields filled in
The profile appears in the profiles list. You can now use it to generate a RunManager.json or launch an execution.

Preview RunManager.json

  1. Click the three-dot menu (⋮) on a profile.
  2. Select Preview RunManager.
  3. A dialog shows the generated RunManager.json content.
  4. Review the configuration to make sure it is correct.

Download RunManager.json

  1. Click the three-dot menu (⋮) on a profile.
  2. Select Download RunManager.
  3. The file downloads to your computer.
  4. Place it in your test project root folder to use it for execution.
Screenshot: RunManager.json preview dialog showing generated configuration

Tips

Start with parallel disabled (1 thread) until your tests are stable. Then gradually increase threads. If tests fail in parallel but pass sequentially, you likely have shared state issues.

Note

Kroviq deduplicates profiles automatically. If you try to create a profile with the same browser + URL + mode + parallel + threads as an existing one, it will warn you about the duplicate.

Import & Sync

Purpose

Import lets you bring an existing Kroviq Core test suite into Studio. It reads your feature files, constants, and test data, then creates the corresponding pages, elements, and test cases in Studio automatically.

When to Use

Use Import when you already have a working test suite built with Kroviq Core and want to manage it visually in Studio. This is a one-time migration step for existing projects.

Step-by-Step Instructions

Import an Existing Test Suite

  1. Make sure your project's Test Suite Path is set correctly (in Project settings).
  2. Navigate to Import & Sync in the sidebar.
  3. Click Import from Test Suite.
  4. Studio scans the test suite folder and shows a summary of what it found:
    • Number of feature files
    • Number of constants classes
    • Number of test data entries
  5. Click Start Import.
  6. Wait for the import to complete. Progress is shown in real-time.
Screenshot: Import progress showing feature files being parsed and linked
Studio creates modules, pages, elements, and test cases from your existing code. The Import History page shows the job with status, duration, and a detailed report.

How Feature Linking Works

During import, Studio parses each step line in your feature files and tries to match elements:

  1. Constant name match — Exact match on the constant name (e.g., TXT_USERNAME)
  2. Fuzzy display name match — If no exact match, tries matching 2+ words from the display name
  3. Null if ambiguous — If multiple elements match, the link is left empty for manual resolution

View Import History

  1. Go to Import & Sync page.
  2. The history table shows all past import jobs.
  3. Click View Report to see detailed results (what was imported, what failed, what needs manual linking).
  4. Use Replay to re-run an import (useful after fixing issues).
  5. Use Delete to remove an import job record.
Screenshot: Import History table with View Report, Replay, and Delete actions
Tips

Run import on a clean project (no existing modules). If you import into a project that already has data, duplicates may be created. Review the import report carefully after each run.

Note

Module tags are auto-generated during import using the naming convention: camelCase splits become UPPER_SNAKE_CASE (e.g., "DemoMUI" → "DEMO_MUI"). RunManager profiles are deduplicated — if an identical profile already exists, it will not be created again.

Execution

Purpose

Execution lets you run your automated tests directly from Studio with one click. You can watch live logs, see real-time progress, and abort if something goes wrong.

When to Use

Use Execution when you want to run tests without opening a terminal. Studio handles everything: building the RunManager, launching Maven, streaming logs, and parsing results.

Step-by-Step Instructions

Launch a Test Execution

  1. Navigate to Execution in the sidebar.
  2. Select the Scope — what to run:
    • Project — Run all tests in the project
    • Module — Run tests for a specific module
    • Test Case — Run a single test case
    • Tags — Run tests matching a tag expression (e.g., @SMOKE)
    • Profile — Use a saved execution profile
  3. If needed, select the specific module, test case, or profile.
  4. Click Run.
  5. The live log panel opens and shows real-time output from the test runner.
Screenshot: Execution page with scope selector, Run button, and live log panel streaming output
Tests start running. The log panel shows live output. When complete, you see a summary with total tests, passed, failed, and skipped counts.

Monitor a Running Execution

While tests are running, you can see:

  • Live log stream — Real-time Maven/Cucumber output via SSE (Server-Sent Events)
  • Elapsed time — Timer showing how long the execution has been running
  • Status indicator — QUEUED → RUNNING → PASSED/FAILED/ERROR

Abort an Execution

  1. While a test is running, click the Abort button.
  2. Confirm the abort in the dialog.
  3. The test process is terminated and status changes to ABORTED.

View Execution History

  1. Navigate to Execution History in the sidebar.
  2. The table shows all past executions with:
    • Status (PASSED, FAILED, ABORTED, ERROR)
    • Pass rate bar (visual indicator)
    • Duration
    • Triggered by (who started it)
  3. Click Replay to re-run the same execution.
  4. Click Monitor to view the log of a past execution.
Screenshot: Execution History table with pass rate bars and action buttons

Execution Statuses

StatusMeaning
QUEUEDExecution is waiting to start
RUNNINGTests are currently executing
PASSEDAll tests passed successfully
FAILEDOne or more tests failed
ABORTEDExecution was manually stopped

Tips

Start with a small scope (single test case or SMOKE suite) to verify your setup works. Once confirmed, run larger suites. Use the Replay button to quickly re-run failed executions after fixing issues.

Warning

Make sure the project's Test Suite Path is set correctly before running. If the path is wrong, the execution will fail with an ERROR status. The path must point to a folder containing a valid pom.xml.

Note

Execution results are parsed from Cucumber output. The pattern "X Scenarios (Y passed)" is used to determine pass/fail counts. Reports are saved in the test project's Reports/ folder.

Reports

Purpose

Reports show you what happened during a test execution — which tests passed, which failed, why they failed, and screenshots of failures. Kroviq generates self-contained HTML and Excel reports automatically.

When to Use

Review reports after every test execution to understand results. Share reports with your team, managers, or stakeholders to communicate test status.

Report Types

ReportFormatBest For
Custom HTML Report.html (self-contained)Detailed step-by-step results with screenshots
Excel Report.xlsx (multi-sheet)Summary data, sharing with non-technical stakeholders
Extent Report.htmlInteractive dashboard with charts and filters

Step-by-Step Instructions

Find Your Reports

  1. After execution completes, reports are saved in your test project: 
    Kroviq_TestSuiteTemplate/Reports/Run_YYYYMMDD_HHMMSS/
  2. Each run creates a new folder with a timestamp.
  3. Inside the folder you will find:
    • TestRun_*_Custom.html — The main HTML report
    • TestRun_*_Report.xlsx — Excel summary
    • Screenshots/ — Failure screenshots
  4. Open the HTML file in any browser to view the report.

Step-by-Step Instructions

Change the Target Environment

  1. Open RunManager.json in your test project.
  2. Change the environmentURL field to the new URL: 
    "environmentURL": "https://staging.wingitairlines.com"
  3. Save the file.
  4. Run your tests — they will now target the new environment.

Enable AI Features

  1. Open RunManager.json.
  2. Add or update the AI feature flags: 
    "rcaEnabled": true,
"defectWriterEnabled": true
  3. Save the file.
  4. On the next execution, failed tests will include RCA analysis and defect drafts in the report.

Configure Wait Times

Kroviq uses tiered waits. You can override defaults in your properties file:

# Wait tier overrides (in seconds)
wait.click=short
wait.type=short
wait.upload=long
wait.page_load=long
TierDefault DurationUse For
SHORT5 secondsQuick actions (click, type)
MEDIUM10 secondsPage transitions, dropdowns
LONG20 secondsFile uploads, page loads, API calls
Tips

Never use Thread.sleep() in your tests. Always use WaitHandler with the appropriate tier. If elements take longer than expected, increase the tier rather than adding hardcoded sleeps.

Note

Settings in RunManager.json override defaults in the framework. If a setting is not specified in RunManager.json, the framework uses its built-in default value.

Troubleshooting

Purpose

Quick solutions to common problems you may encounter when using Kroviq. Each issue includes symptoms, cause, and step-by-step fix.

Build Failures

Issue: "BUILD FAILURE" during mvn clean install

Symptoms: Maven build fails with dependency resolution errors or SSL certificate errors.

Cause: Network issues, proxy settings, or SSL certificate validation failures.

Solution:

  1. Try adding SSL bypass flags: 
    mvn clean install -Dmaven.wagon.http.ssl.insecure=true -Dmaven.wagon.http.ssl.allowall=true -Dmaven.resolver.transport=wagon
  2. If behind a corporate proxy, configure Maven's settings.xml with proxy details.
  3. Delete the .m2/repository cache and rebuild.

Issue: "Java version not supported"

Symptoms: Compilation errors mentioning unsupported class file version.

Cause: You are using Java 11 or lower. Kroviq requires Java 17+.

Solution:

  1. Check your Java version: java -version
  2. Install Java 17 or higher.
  3. Set JAVA_HOME to point to the Java 17 installation.
  4. Restart your terminal and IDE.

Execution Failures

Issue: "WebDriverException: Cannot find browser binary"

Symptoms: Test fails immediately with a driver error.

Cause: Browser is not installed, or WebDriverManager cannot download the driver.

Solution:

  1. Make sure the browser specified in RunManager.json is installed on your machine.
  2. Check your internet connection (WebDriverManager downloads drivers automatically).
  3. If offline, manually place the driver binary in your system PATH.

Issue: "NoSuchElementException" or "Element not found"

Symptoms: Test fails because it cannot find an element on the page.

Cause: The locator is wrong, the page has not loaded yet, or the UI has changed.

Solution:

  1. Open the application manually and verify the element exists.
  2. Use browser DevTools (F12) to check if the locator matches.
  3. If the UI changed, update the locator in the Object Repository.
  4. If it is a timing issue, check that the correct wait tier is being used.

Issue: "StaleElementReferenceException"

Symptoms: Test finds the element but fails when trying to interact with it.

Cause: The page refreshed or re-rendered between finding and clicking the element.

Solution:

  1. This is usually handled automatically by Kroviq's retry mechanism.
  2. If it persists, add a wait before the action (use WaitHandler, not Thread.sleep).
  3. Check if the page has dynamic content that reloads frequently.

Issue: "ElementClickInterceptedException"

Symptoms: Click action fails because another element is covering the target.

Cause: A modal, overlay, loading spinner, or toast is blocking the element.

Solution:

  1. Kroviq automatically retries with a JavaScript fallback click.
  2. If it still fails, wait for the overlay to disappear (use LoadingStateDetector).
  3. Check if a toast notification is covering the button.

Studio Issues

Issue: Studio backend won't start

Symptoms: java -jar kroviq-studio-api-1.0.0.jar fails or port 8080 is in use.

Cause: Another process is using port 8080, or the H2 database file is locked.

Solution:

  1. Check if port 8080 is in use: netstat -ano | findstr :8080
  2. Kill the conflicting process or change the port in application.properties.
  3. If the DB is locked, close all other Studio instances and try again.

Issue: Execution shows ERROR status immediately

Symptoms: Clicking Run shows ERROR without any log output.

Cause: The Test Suite Path in the project settings is wrong or the folder does not contain a pom.xml.

Solution:

  1. Go to Project settings and verify the Test Suite Path.
  2. Make sure the path uses forward slashes (e.g., D:/Kroviq_Framework/Kroviq_TestSuiteTemplate).
  3. Verify the folder contains a valid pom.xml file.

Issue: Tests pass locally but fail in parallel

Symptoms: Tests work with 1 thread but fail with multiple threads.

Cause: Shared mutable state between tests (static variables, shared browser sessions).

Solution:

  1. Make sure all test state is stored in TestContext (which is thread-safe).
  2. Never use static mutable variables in step definitions.
  3. Each thread gets its own WebDriver instance — do not share drivers.

Issue: Import creates duplicate modules

Symptoms: After import, you see duplicate modules or elements.

Cause: Import was run on a project that already had data.

Solution:

  1. Delete the duplicates manually.
  2. For future imports, use a clean project (no existing modules).
  3. Or delete all modules first, then re-import.
Tips

When reporting issues, always include: (1) the error message, (2) the step that failed, (3) the locator being used, and (4) a screenshot of the application state. This helps resolve issues much faster.

Best Practices

Purpose

Follow these guidelines to keep your test automation maintainable, reliable, and scalable as your project grows.

Locator Strategy

Choose locators in this priority order (most stable first):

  1. data-testid — Best choice. Dedicated test attribute that never changes with UI redesigns.
  2. id — Good if unique and stable. Avoid auto-generated IDs.
  3. name — Good for form fields.
  4. CSS selector — Shorter and faster than XPath for simple cases.
  5. XPath — Use as last resort. Keep it short (max 3 levels deep).
Tips

Ask your developers to add data-testid attributes to key elements. This makes locators immune to CSS class changes and UI refactoring.

Test Data Management

  • Choose ONE data source (JSON or Excel) per project and stick with it.
  • Never store real credentials in test data files.
  • Use meaningful key names that describe the data (e.g., "ValidEmail" not "Data1").
  • Keep test data files small — one file per module.
  • Use the TD_ prefix consistently in all feature files.

Parallel Execution

  • Start with 1 thread until all tests pass reliably.
  • Increase threads gradually (2 → 4 → 8).
  • Each test must be independent — no test should depend on another test's result.
  • Use unique test data per test case to avoid conflicts.
  • Never share WebDriver instances between threads.

Maintenance

  • Review and update locators when the UI changes.
  • Delete unused elements from the Object Repository.
  • Keep feature files short (5–15 steps per scenario).
  • Run the full regression suite at least once per sprint.
  • Fix flaky tests immediately — do not ignore them.
  • Use AI RCA reports to identify patterns in failures.
Note

The Kroviq framework handles many common issues automatically (stale element retry, JS fallback clicks, self-healing locators). If you follow these best practices, most tests will be stable and maintainable with minimal effort.
