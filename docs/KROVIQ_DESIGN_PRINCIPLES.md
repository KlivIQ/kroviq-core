# Kroviq Design Principles

## 1. Product Philosophy

Kroviq Studio is a **Low-Code Test Design Platform**.

It is not: - An admin dashboard - A reporting tool - A project
management application - A code editor

Users should feel they are **designing automation**, not managing
records.

------------------------------------------------------------------------

## 2. Brand Personality

Kroviq should feel: - Professional - Precise - Engineered - Calm -
Trustworthy - Enterprise-grade - Modern without chasing trends

Never playful. Never futuristic. Never "AI magic."

------------------------------------------------------------------------

## 3. Visual Language

Inspired by: - Linear - Raycast - GitHub Desktop - Figma - JetBrains
IDEs

Not inspired by: - AI websites - Crypto dashboards - Startup landing
pages - Analytics dashboards - Material UI templates

------------------------------------------------------------------------

## 4. Design Keywords

-   Precision
-   Structure
-   Workflow
-   Control
-   Confidence
-   Simplicity

------------------------------------------------------------------------

## 5. Colour Philosophy

Primary accent: - Copper / Terracotta

Supporting colours: - Warm graphite - Warm greys - White

Use status colours only for Success, Warning and Error.

------------------------------------------------------------------------

## 6. Never Use

-   Gradients
-   Glassmorphism
-   Neon colours
-   Heavy glows
-   AI brain icons
-   Circuit graphics
-   Decorative animations

------------------------------------------------------------------------

## 7. Layout Principles

-   One primary action per screen.
-   Controlled density.
-   Intentional whitespace.
-   Reusable layouts.

Every screen answers: - Where am I? - What am I working on? - What
should I do next?

------------------------------------------------------------------------

## 8. Components

Everything should be reusable. Never build one-off UI.

------------------------------------------------------------------------

## 9. Typography

-   Large headings
-   Small uppercase section labels
-   Monospace only for IDs, tags, paths, assets and technical values.

------------------------------------------------------------------------

## 10. Dashboard Philosophy

The dashboard is a launch point.

Not analytics. Not reporting.

It answers: - Where am I? - What was I doing? - What needs attention? -
Where do I continue?

------------------------------------------------------------------------

## 11. Navigation Philosophy

Project → Module → Page → Object Repository → Test Case → Generated
Assets

------------------------------------------------------------------------

## 12. Empty States

Teach users what the page is for and what to do next.

------------------------------------------------------------------------

## 13. Generated Assets

Generated Assets are a product capability, not a download list.

Always show: - Source - Sync status - Generation time - Regeneration
availability

------------------------------------------------------------------------

## 14. AI Philosophy

AI assists.

Users stay in control.

Everything generated should be transparent, editable and explainable.

------------------------------------------------------------------------

## 15. Commercial Standard

Every screen should make users think:

> "This looks like a serious engineering product."

Not:

> "This looks like an internal company tool."

------------------------------------------------------------------------

## Guiding Principle

**Design automation. Stay in control.**
